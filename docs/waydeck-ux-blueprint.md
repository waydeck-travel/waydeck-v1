### 1.2 Navigation Pattern

| Approach | Recommendation |
|----------|----------------|
| **Router** | `go_router` with nested routes |
| **Main Navigation** | Single-stack push/pop (no bottom tabs for MVP) |
| **Trip Context** | Trip Overview as the hub; Timeline is a scrollable section within it |

**UX Assumption:** MVP uses a simple stack-based navigation without bottom tabs. A drawer or top-right profile icon provides access to Settings/Profile.

## 6. UX Assumptions & Decisions

The following assumptions are made where the spec is silent or ambiguous:

| # | Assumption | Rationale |
|---|------------|-----------|
| 1 | **No bottom navigation tabs** in MVP | Spec implies a simple single-stack flow; trips are the main content |
| 2 | **Drawer for settings access** | Common pattern for secondary nav; keeps focus on trip content |
| 3 | **Timeline is part of Trip Overview screen** (not a separate screen) | Spec section 6.3 describes them together; reduces navigation depth |
| 4 | **FAB for adding items** shows action sheet to select type | More discoverable than 4 separate buttons; reduces clutter |
| 5 | **Documents shown as horizontal thumbnail grid** in detail screens | Allows quick visual scanning; common mobile pattern |
| 6 | **Layover shown only between consecutive transports with matching cities** | Per spec section 2.4; avoids false positives |
| 7 | **Notes don't have documents attached** | Spec only mentions documents for trips and trip items with tickets/vouchers |

|--------|-------------------|

| Error Type | Display Method |
|------------|----------------|

| Auth failure | Inline error message below form |
| Validation error | Inline error on specific field |
| Not found | Full screen error with back button |
| Upload failure | `TDToast` + retry option |

## 9. Accessibility Considerations

- All interactive elements should have semantic labels
- Minimum touch target of 48x48 dp
- Color contrast meets WCAG AA standards
- Timeline dots should have text alternatives (screen reader support)
- Forms should support keyboard navigation

## 10. Summary

This UX Blueprint defines:

1. **17 screens** covering auth, trip management, timeline, item details, forms, and documents
2. **Navigation structure** using `go_router` with stack-based routing
3. **Component mapping** to TDesign Flutter widgets
4. **Custom Timeline widget** design using TDesign primitives + minimal custom widgets
5. **Wireframe descriptions** in ASCII format for implementation guidance
6. **14 documented UX assumptions** filling gaps in the spec

The Flutter App Agent can use this blueprint to implement screens without guessing at layout or navigation patterns.
